# 정책 노트

스키마(데이터 구조) 차원의 결정은 `schema-note.md`에 두고, 이 문서는 런타임 동작 정책(재시도, 백오프, 워커 주기, stuck 임계 등)에 대한 결정을 누적함.

## 결정 1. 수동 재시도 시 카운터 미리셋 (보수적 정책)

**결정:** `DEAD_LETTER` 상태의 알림을 운영자가 수동으로 재시도시킬 때, `fail_count`와 `stuck_count`를 **초기화하지 않음.** 상태만 `PENDING`으로 되돌리고 `retry_after`만 비움.

**동작 결과:** 워커가 다시 픽업하여 처리할 때, 단 한 번이라도 실패하면 `fail_count + 1 >= max_fail` 조건에 즉시 걸려 다시 `DEAD_LETTER`로 떨어짐. 즉, 수동 재시도는 사실상 "한 번의 추가 시도 기회 부여"의 의미를 가짐.

**사유:**
- 알림이 데드레터에 도달했다는 것은 이미 정상 재시도 사이클을 모두 소진했다는 의미. 카운터를 리셋해 풀 사이클을 다시 부여하면, 운영자가 근본 원인 분석 없이 재시도 버튼만 반복적으로 누르는 안티패턴을 유발할 수 있음.
- 일시 장애로 데드레터에 빠진 케이스라면 한 번의 추가 시도로도 충분히 복구 가능. 그렇지 않은 케이스(데이터 자체 문제, 외부 시스템 영구 장애 등)는 카운터 리셋해도 어차피 다시 실패함.
- 알림의 lifetime 전체 실패 횟수가 필요한 경우 `notification_error_log` 테이블에서 `COUNT(*)`로 정확히 조회 가능. `fail_count`는 "현재 사이클의 잔여 재시도 예산"으로 의미를 좁혀 사용함.

**대안:** 만약 운영 중 "근본 원인을 해결한 후 풀 사이클 재시도"가 필요한 경우가 빈번해지면, 별도 API(`force-retry-with-reset`)로 카운터 초기화 옵션을 추가하는 방향으로 확장 가능. 현 시점에서는 단일 정책으로 단순하게 유지.
