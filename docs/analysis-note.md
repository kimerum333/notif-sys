# 과제 문서 분석
본 문서는, 이번 과제를 정독하며 사고한 흔적을 메모한 문서이다.

## 트랜잭션의 분리
- 알림 처리 실패가 비즈니스 트랜잭션에 영향을 주면 안되며, 이를 예외 무시로 달성해선 안 된다.
- AI(claude)와 대화한 결과, 해당 요구사항은 outbox 패턴으로 해결하는 것이 일반적인 해결책임을 확인.
- 그러나, outbox 패턴을 학습한 결과, 이번 과제에서 비즈니스 트랜잭션은 구현 범위 밖이므로 변형이 필요.
- 알림 발송 요청은 API로부터 외부 시스템에 의해 들어온다는 전제이므로, 비즈니스 시스템에서 비즈니스 트랜잭션을 커밋한 뒤 알람 API에 요청을 하는 흐름인 건가?
- 이 경우, 비즈니스 트랜잭션 커밋 후 알람 API요청 사이에 장애가 발생할 경우 알림이 오지 않는 문제가 있을 수 있다.
- 그런데, 외부 시스템인 비즈니스 시스템 쪽에 비즈니스로직 성공시 알림발송API를 확실히 요청해야할 책임이 있으므로 이번 과제에서 고려해야 할 바는 아님.
- 하지만, 이 전제 위에서 생각할 경우 '알림 처리 실패가 비즈니스 트랜잭션에 영향을 줄' 가능성이 아예 없는데, 어디에 주의해서 풀어야 하는 것인지를 잘 모르겠다. 설마, 알림 발송 실패시 비즈니스 로직을 롤백시키는 요청을 역으로 쏘는 식으로 설계하지 말라는 의미인가?
- 일단은, 'API요청' 자체를 비즈니스 로직이라고 생각하고 알림 발송과 요청 접수의 트랜잭션을 분리하라는 요구사항인 것으로 가닥을 잡는다.

## 알람의 영속화
- 알림 발송 요청 API의 결과는 요청 접수 완료. DB에 영속화하는 것으로 끝.
- 이후, 특정알림 / 특정 수신자 기준 알림 목록 조회 기능이 있는데... 관리자 시스템까지 염두에 두는 것 같지만, UI는 이번 과제 대상이 아니니 넘기고, 수신자 기준이 있으니 인덱스 하나 잡긴 해야 할 듯.
- 실패 사유가 기록되어야 하며, 재시도 정책을 고려해야 한다... 알림 테이블에 Status외에도, 실패 횟수, 재시도 가능 시각 등을 넣으란 의미로 보임.

## 워커 설계
- api는 접수 완료로 라이프사이클 끝.
- 즉, 주기적으로 알림 테이블을 셀렉트치며 알림을 발송하는 워커가 있어야 한다는 의미.
- 다중 인스턴스 환경에서도 동일 알림이 중복 처리되어선 안 된다. 동시성. 워커 간 경쟁 상태를 제어하기 위한 락이 필요.
- 솔직히, 여기서 스스로 해결 방법을 떠올리지 못했다. claude 와의 대화로, 단일 진실의 원천인 DB에 락을 걸 방법이 있다는 것을 확인.

- select 시 for update 를 걸어 셀렉트와 동시에 락을 걸 수 있는 기능을 학습.
- 락의 스코프는 해당 table에서 select 된 각 Row 인 것으로 확인.
- 이후, skip locked 옵션을 통해, 다른 워커는 락이 걸리지 않은 row 들을 select 할 수 있다.
- 중간에 새로운 알림이 생성되어도, 이는 새로운 row 의 record 이므로 다른 워커가 접근 가능.
- 따라서, 전체 서비스의 퍼포먼스에는 문제가 없을 것이라 판단해 채용.

## 멱등성 보장
### 요청 접수 시
- FE에서 스로틀링 등을 동원한다 해도, 네트워크 상태에 따라 얼마든지 동일 요청이 중복으로 들어올 수 있다.
- 가장 단순한 해결책은 FE에서 멱등성 키를 발급해 요청API에 포함하도록 하는 것.
- 그러나 이는 FE에 부담을 주는 일이라 조금 꺼려졌고, 더 좋은 해결책이 없는지 claude 와 상담.
- ...허무하게도, 과제에 주어진 수신자ID와 참조 데이터(이벤트Id,강의Id)를 복합PK(또는, 인조키+복합 유니크 제약)으로 지정하는 시점에서 이미 DB단에서 멱등성이 보장된다는 답변을 얻었다.
- 왜 이걸 생각 못했을까. 별도로 처리할 부분 없음.

### 알림 발송 시
- 윗 문단의 워커 설계시에 이미 고려 완료.

### 읽음 처리 시
- 추가 점수에서, 복수의 기기에서 동시에 읽음 처리 요청이 왔을 때 어떻게 처리할지 고민해보라는 항목이 있는데...
- 알림의 상태를 update 하는 것뿐이라 고민할 이유가 없지 않나?

## 상태 관리와 재시도 정책
- 유한상태기계. 알람의 상태를 정의하고, 전이 조건을 설계한다.
- 실제 이메일 발송이 불필요 + 인앱 알림도 UI가 없어 실패가 불가능. 실패를 위한 무언가를 만들어야 한다.

### 상태관리 Happy path
1. pending : 발송 대기. API요청으로 알림이 갓 만들어진 직후.
2. processing : 처리 시작. 워커가 select 로 긁어온 직후 update.
3. sent : 알림 발송 상태. 워커가 성공적으로 알림을 발송한 직후. 아직 읽히지 않은 알림.
4. read : 읽힘. API로 읽음 처리 요청을 받은 직후.
### 상태관리 Error case
- pending 상태에서는 문제의 소지가 없는 것으로 보인다.
#### Processing 도중에 워커 사망
- 처리 중 상태가 일정 시간 이상 지속되는 경우 복구 방법을 설계하라. 는 요구사항에 해당.
- DB는 살고 워커만 죽으면 for update 로 건 락은 남을까 고민했지만, 락은 트랜잭션에 종속되므로 워커가 죽으면 락은 풀린다고 한다.
- processing 상태인 레코드의 updated at 과 sysdate 를 비교해서 재처리 대상 선별 가능. 추가 칼럼 필요 없어보였지만...
- 재처리 중 워커가 또 죽으면 updated at 이 갱신되지 않는다. stuck_count 같은 칼럼을 만들어 1씩 올리며 처리하자.
#### Processing 도중에 에러 발생
- 일단, email 발송요청, 인앱알람 발송요청 등은 network bound.
- 성능 고려하면 비동기가 기본 전제일 것.
- 실패 응답은 죽어있던 코루틴이 깨어나서 받은 뒤 update 해야 할 것.=> (클로드 도움으로 교정)이건 파이썬식 사고. Java에선 콜백 체이닝방식.
##### 멱등성 이슈
- 알림발송 요청 후 응답을 받기 전에 워커가 죽어버리면, 알림은 processing 상태에서 stuck.
- processing 상태에서 멈춘 작업을 재처리하는 로직이 돌 경우, 알림이 한번 더 가는 문제가 있다.
- 이 시스템에서 알림의 식별과 그로 인한 멱등성 보장은 DB가 보장하는 바, 외부 시스템은 자력으론 멱등성을 보장할 수 없다. 멱등성 키를 보내주고, 외부 시스템에서 멱등성 키를 이용한 중복 처리 방지 기능이 존재하길 바라야 한다.

### 재시도 정책.
- 재시도 횟수가 늘어날 때마다, 다음 재시도 간격을 늘리는 것이 좋다고 알고 있다.
- retry_after 같은 칼럼을 만들어서, fail_count 칼럼을 참조해 다음 재시도가 가능한 시각을 기록하는 식으로 하자.
- 별도의 폴링 스레드를 구성해야 할 테고, 이 또한 기본 알림 발송과 마찬가지로 for update/skip locked 적용.
- 또한, 명시적으로 retry_after 이 없는 레코드여도, updated at이 한참 예전인 processing 등을 fail로 잡아내는 select 문이 중요.

## 외부 리소스 관리
- 알림 타입별 메시지는, 외부 json파일 등을 읽어오는 방향으로 짜면 개발자의 코드수정 없이 원활한 운영이 가능할듯.
- 이런 식으로 재시도 횟수 정책 같은 것도 외부 config 로 주입받으면 운영시 편리하지 않을지?

- 운영환경으로 전환하면, cdc 로 notification 테이블의 변경사항을 추적하고, 변경시 메시지 브로커에 메시지를 전달하는 식으로 바뀐다.
- 나머지 요소들은 변경할 필요 없이, 폴링 모듈만 교체하는 것으로 전환 가능.

## AI의 첨삭 결과

### 잘 짚은 부분
- **`SELECT FOR UPDATE SKIP LOCKED`** — 정확함. 다중 인스턴스에서 중복 처리 막는 가장 현실적인 방법이고, 과제 수준에서 기대하는 답안.
- **복합 유니크 제약으로 멱등성 보장** — FE에 부담을 준다고 느낀 감각도 맞고, DB 제약이 더 단단한 보장. 결론 정확.
- **stuck_count + updated_at 기반 복구** — retry_after 칼럼으로 재시도 시각 관리하는 것도 맞는 방향.
- **비즈니스 트랜잭션 분리 해석** — 결론이 맞음. "API 접수 자체를 비즈니스 로직으로 보고, 발송과 트랜잭션을 분리"가 과제가 의도한 바. Outbox 패턴의 핵심은 "DB에 먼저 쓰고, 처리는 나중에"인데, 이번 과제에서 알림 테이블 자체가 outbox 역할을 함.

### 보완이 필요한 부분

#### 1. 워커가 실제로 몇 개인지 명확히 안 나옴
분석 내용이 흩어져 있는데, 정리하면 세 가지 역할이 필요함:
- **메인 워커**: `PENDING` → `PROCESSING` → `SENT/FAILED`
- **재시도 워커**: `FAILED` + `retry_after <= now()` + `fail_count < max` → 재처리
- **복구 워커**: `PROCESSING` + `updated_at`이 오래된 것 → `PENDING` 또는 `FAILED`로 되돌리기

이 세 역할을 하나의 스케줄러 안에서 순서대로 처리하는 게 구현이 단순하고 이 과제 수준에선 충분함.

#### 2. 알림 타입 정의가 없음
과제 문서에 "수강 신청 완료, 결제 확정, 강의 시작 D-1, 취소 처리" 예시가 있는데, 이를 enum으로 어떻게 정의할지가 빠져 있음. 템플릿 관리(선택 구현)를 붙이려면 타입 정의가 선행되어야 함.

#### 3. 읽음 처리 멱등성 — 분석 결론은 맞음
"고민할 이유가 없지 않나?" → 맞음. 이미 READ인 알림에 READ 요청이 오면 그냥 no-op. 동시에 두 기기에서 READ 요청이 와도 둘 다 `UPDATE SET status = 'READ' WHERE id = ?`를 실행하면 한 놈은 1 row affected, 나머지는 0 row affected, 결과는 동일. 별도 처리 불필요하다고 문서화만 하면 되는 항목.

#### 4. 발송 중 멱등성 이슈 — 과제 범위에서는 신경 덜 써도 됨
"처리 중 워커 사망 → 재처리 시 중복 발송" 고민은 맞지만, 과제가 Mock 발송이라 실제 중복 발송은 없음. 외부 시스템에 멱등성 키를 넘기는 구조를 코드 레벨에서 인터페이스로만 표현해두는 걸로 충분함. 이 고민을 기록해둔 것 자체는 채점자에게 좋은 인상을 줄 수 있음.
